# Обработка ошибки 429 (Too Many Requests)

## Описание проблемы
WooCommerce REST API может возвращать статус `429` при лимите заказов. Эта ошибка требует специальной обработки, чтобы не разрушить состояние приложения.

## Решение
Добавлена корректная обработка ошибки 429 в компоненте [Checkout.tsx](src/components/Checkout.tsx).

### Что было сделано:

#### 1. **Добавлено состояние для сообщения об ошибке**
```tsx
const [errorMessage, setErrorMessage] = useState<string>('');
```
Это позволяет отдельно управлять сообщением об ошибке 429 от ошибок RTK Query.

#### 2. **Обновлена функция `handleConfirmOrder`**
Добавлена проверка статуса ошибки:
- **Проверка**: `err.status === 429`
- **Сообщение**: Берется из `err.data?.message` или использует стандартный текст
- **Поведение**:
  - ✅ Показывает сообщение об ошибке пользователю
  - ✅ Корзина **не очищается**
  - ✅ Модальное окно **остается открытым**
  - ✅ Состояние `isSubmitting` устанавливается в `false`
  - ✅ Стандартная ошибка сервера **не показывается**
  - ✅ Форма **не ломается**
  - ✅ Пользователь может попробовать снова позже

#### 3. **Обновлен интерфейс модального окна**
- Добавлено отображение сообщения об ошибке в желтой коробке (warning style)
- Кнопки теперь отключаются при `isSubmitting = true`
- Кнопка подтверждения показывает спиннер загрузки и текст "Создание..." во время отправки

#### 4. **Обновлена основная форма**
Ошибка из RTK Query показывается только если **не** есть `errorMessage`:
```tsx
{error && !errorMessage && (
  <div>Ошибка при создании заказа...</div>
)}
```
Это предотвращает показ стандартной ошибки, когда уже показано сообщение о лимите.

#### 5. **Очищение ошибки при отмене**
В функции `handleCancelConfirm` ошибка очищается:
```tsx
setErrorMessage('');
```

# Отображение способа доставки в чеке

## Проблема
После создания заказа клиент может забыть, какой способ доставки он выбрал (самовывоз или доставка).

## Решение
Добавлено четкое отображение способа получения заказа в [OrderReceipt.tsx](src/components/OrderReceipt.tsx).

### Что было добавлено:

#### 1. **Функция для получения типа заказа**
```tsx
const getOrderType = () => {
  const orderTypeData = currentOrderData.meta_data?.find((m: any) => m.key === 'order_type');
  return orderTypeData?.value || 'delivery';
};
```

#### 2. **Выделенный блок со способом получения**
Большой, яркий блок с:
- **Эмодзи**: 🛍️ для самовывоза, 🚗 для доставки
- **Цвета**: зеленый для самовывоза, синий для доставки
- **Размер текста**: 2xl, жирный шрифт для лучшей видимости
- **Позиция**: прямо после информации о клиенте (первое, что видит пользователь)

#### 3. **Дополнительное отображение в "Информации о заказе"**
Также способ получения показывается в разделе с метаинформацией:
```
Способ получения: 🛍️ Самовывоз
или
Способ получения: 🚗 Доставка
```

## Где видит клиент

1. **Основной блок** (самый заметный):
   - Сразу после информации о клиенте
   - Занимает всю ширину экрана
   - С иконкой и большим текстом

2. **В разделе "Информация о заказе"**:
   - Под методом оплаты
   - С адресом доставки

3. **При печати чека**:
   - Все элементы сохраняются при print()

## Поток обработки ошибки 429

```
Пользователь нажимает "Оформить заказ"
         ↓
Открывается модальное окно подтверждения
         ↓
Пользователь нажимает "Подтвердить"
         ↓
setIsSubmitting(true) → показывается спиннер
setErrorMessage('') → очищается предыдущая ошибка
         ↓
Отправляется запрос к WooCommerce API
         ↓
Возвращается статус 429
         ↓
В catch блоке проверяется err.status === 429
         ↓
Если ДА:
  - Извлекается сообщение из err.data.message
  - setErrorMessage(message) → ошибка показывается в модале
  - setShowConfirmModal(true) → модаль остается открытой
  - return → выход из функции (не выполняются удаления данных)
         ↓
setIsSubmitting(false) → спиннер исчезает, кнопки снова активны
         ↓
Пользователь видит сообщение об ошибке и может попробовать снова
```

## Гарантии

- ✅ **Корзина не очищается** - `clearCart()` не вызывается
- ✅ **Модаль не закрывается** - `setShowConfirmModal(true)` переоткрывает её
- ✅ **isSubmitting = false** - в блоке `finally` после всех операций
- ✅ **Стандартная ошибка не показывается** - проверка `!errorMessage` в условии
- ✅ **Форма не ломается** - никакие состояния формы не изменяются
- ✅ **Можно попробовать снова** - кнопка "Подтвердить" остается активной
- ✅ **Способ доставки видно в чеке** - отображается четко и ярко

## Примеры сообщений об ошибке

Типичные сообщения от WooCommerce:
- `"Too many orders. Please try again later."`
- `"Request limit exceeded. Please wait before creating another order."`
- Или кастомное сообщение из административной панели

## Тестирование

Для тестирования ошибки 429:
1. Откройте DevTools (F12)
2. Перейдите на вкладку Network
3. Создавайте несколько заказов подряд
4. Когда прилетит 429, проверьте:
   - Видно ли сообщение об ошибке в модале
   - Остается ли модаль открытой
   - Очищается ли корзина (НЕ ДОЛЖНА!)
   - Можно ли попробовать снова

Для тестирования отображения способа доставки:
1. Создайте заказ с **самовывозом** (самовывоз)
2. Проверьте, что в чеке:
   - Видно 🛍️ Самовывоз
   - Зеленая подсветка блока
3. Создайте заказ с **доставкой**
4. Проверьте, что в чеке:
   - Видно 🚗 Доставка
   - Синяя подсветка блока
5. Попробуйте распечатать оба варианта (Print)

## Технические детали

### Обработка 429
- **Файл**: `src/components/Checkout.tsx`
- **Функция**: `handleConfirmOrder`
- **RTK Query**: Используется `.unwrap()` для лучшей обработки ошибок
- **Тип ошибки**: `err.status === 429`
- **Структура ошибки**: `{ status: 429, data: { message: "..." } }`

### Отображение способа доставки
- **Файл**: `src/components/OrderReceipt.tsx`
- **Функции**: 
  - `getOrderType()` - получает тип из meta_data
  - `getOrderTypeDisplay()` - возвращает текст с эмодзи
- **Данные хранятся**: В `meta_data` заказа WooCommerce
- **Ключ**: `order_type` (значения: `pickup` или `delivery`)

# Ограничение активных заказов (2 одновременно)

## Описание проблемы
Нужно запретить клиентам создавать более 2 активных заказов одновременно с одного IP адреса.

## Решение
Добавлена проверка количества активных заказов перед отправкой нового заказа.

### 📋 Что было сделано:

#### 1. **Добавлен новый API endpoint** в [publicApi.ts](src/app/services/publicApi.ts)

```typescript
checkActiveOrdersCount: builder.query({
  query: () => `wc/v3/orders?per_page=100&status=pending,on-hold,processing`,
  providesTags: ['Orders'],
}),
```

**Что делает:**
- Получает список заказов со статусами: `pending`, `on-hold`, `processing`
- Возвращает только активные заказы (не завершенные, не отмененные)

#### 2. **Добавлена проверка в handleConfirmOrder** в [Checkout.tsx](src/components/Checkout.tsx)

```typescript
// Проверяем количество активных заказов с этого IP
if (activeOrdersData && activeOrdersData.length >= 2) {
  setErrorMessage('У вас уже есть 2 активных заказа. Дождитесь их выполнения перед созданием нового.');
  setShowConfirmModal(true);
  console.warn('Too many active orders:', activeOrdersData.length);
  return;
}
```

**Логика:**
- Перед отправкой заказа проверяем количество активных
- Если `>= 2` - показываем ошибку в модальном окне
- Если `< 2` - разрешаем создать новый заказ

#### 3. **Хук для проверки**

```typescript
const { data: activeOrdersData } = useCheckActiveOrdersCountQuery(undefined, {
  pollingInterval: 0, // Проверяем только по требованию
});
```

## Сообщения об ошибках

**Когда уже 2+ активных заказов:**
```
"У вас уже есть 2 активных заказа. Дождитесь их выполнения перед созданием нового."
```

- Отображается в желтой коробке в модальном окне подтверждения
- Модаль остается открытой
- Пользователь может закрыть модаль и попробовать позже

## Технические детали

### Статусы "активных" заказов:
- `pending` — в ожидании обработки
- `on-hold` — на удержании
- `processing` — обрабатывается

### Статусы "не активных":
- `completed` — завершены ✓
- `cancelled` — отменены ✗
- `refunded` — возвращены
- `failed` — ошибка

## Важные замечания

### ⚠️ IP-адресация на backend
Проверка работает корректно, если на backend'е:
- WooCommerce видит реальный IP клиента
- Может быть настроена проксирование (X-Forwarded-For headers)

### 🔐 Безопасность
- IP-адрес передается в заголовках WooCommerce
- Фронтенд не может подделать IP (это свойство HTTP запроса)
- Бэкенд должен правильно парсить IP из headers

### 🔄 Кеширование
- Запрос выполняется **только по требованию** (pollingInterval: 0)
- При отправке заказа проверка выполняется один раз
- Для свежих данных используйте RTK Query методы invalidatesTags

## Тестирование

### Сценарий 1: Блокировка при 2 активных
1. Создайте заказ (статус: on-hold) → успешно ✓
2. Создайте еще один заказ (статус: on-hold) → успешно ✓
3. Попытайтесь создать 3-й заказ → **ЗАБЛОКИРОВАНО** ✗
   - Видно сообщение об ошибке
   - Можно закрыть модаль и попробовать позже

### Сценарий 2: Разблокировка после выполнения
1. У вас 2 активных заказа (заблокировано)
2. Ресторан обновляет статус первого на `completed` ✓
3. Теперь активных только 1 → можно создать новый ✓

### Сценарий 3: Разблокировка после отмены
1. У вас 2 активных заказа (заблокировано)
2. Отменяете один заказ (статус: `cancelled`)
3. Теперь активных только 1 → можно создать новый ✓
